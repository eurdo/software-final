## 一、需求管理說明書(SRS)  
### 1.1 系統目的  

本物流資訊管理系統之主要目的在於整合物流作業流程，包含訂單管理、倉儲管理與配送追蹤，透過資訊系統取代傳統人工處理方式，以提升作業效率、降低錯誤率，並提供即時且正確的物流資訊。系統可協助管理人員即時掌握物流狀況，強化整體營運管理能力。  

### 1.2 使用者角色

本系統依使用者職責區分不同角色，並賦予相應之操作權限，說明如下：

•系統管理員  
 系負責系統帳號管理、使用者權限設定及系統基本資料維護。  

•物流管理人員  
 物負責訂單管理、物流狀態監控、庫存查詢與報表分析。  

•倉儲人員  
 倉負責商品入庫、出庫及庫存資料更新。  

•配送人員  
 配負責配送狀態回報與運送資訊更新。 

### 1.3 功能需求  

本系統需提供下列主要功能：  

•使用者登入與權限管理  
 使不同角色使用者可依其權限存取系統功能。  

•訂單管理功能  
 訂包含訂單建立、查詢、修改與狀態更新。  

•庫存管理功能  
 庫提供商品庫存查詢、入庫與出庫紀錄管理。  

•物流配送管理功能  
 物記錄配送流程並即時更新配送狀態。  

•報表與查詢功能  
 產生物流相關統計報表，供管理決策參考。  
1.4 非功能需求  
為確保系統穩定與品質，需符合以下非功能性需求：  

•系統效能  
 系系統應於合理時間內完成資料查詢與處理，避免操作延遲。  
 
•系統安全性  
 使用者須經身份驗證後方可使用系統，並確保資料存取安全。  

•系統穩定性  
 系統需能長時間穩定運作，並具備錯誤處理機制。  

•使用者介面  
 操作介面需簡潔易懂，降低使用者學習成本。

## 二、概要設計說明書(HDL)  
### 2.1 系統架構  

本物流資訊管理系統採用分層式系統架構設計，主要可分為「使用者介面層」、「應用系統層」與「資料庫層」三個層級。此架構可有效區分系統功能，提升系統維護性與擴充性。  

使用者介面層負責提供系統操作畫面，讓不同角色的使用者可透過網頁介面進行系統操作。應用系統層則負責處理各項業務邏輯，包括訂單處理、庫存管理與物流配送流程。資料庫層用於儲存系統所需之各項資料，如使用者資料、訂單資料、庫存資料及物流狀態紀錄。  

透過此分層架構設計，可降低系統模組間的相依性，使系統更具穩定性與彈性。

### 2.2 模組規劃  

本系統依功能需求規劃多個模組，各模組職責分明，說明如下：  

• 使用者管理模組：  
 負責使用者帳號建立、登入驗證及權限控管。  

• 訂單管理模組：  
提供訂單建立、查詢、修改及訂單狀態管理功能。  

• 庫存管理模組：  
負責商品入庫、出庫及庫存數量更新與查詢。  

• 配送管理模組：  
記錄配送流程，並即時更新配送狀態。  

• 報表分析模組：  
彙整系統資料並產生統計報表，供管理人員決策參考。  

透過上述模組化規劃，系統可依實際需求進行功能擴充或模組調整，提升整體設計彈性。  


## 三、詳細設計說明書(LLD)  
### 3.1 功能說明  

本系統依概要設計之模組規劃，將各功能進一步細化，明確定義其處理流程與系統行為，說明如下：  

• 使用者管理功能  
系統提供使用者登入、登出與身分驗證機制，並依不同角色賦予對應操作權限，確保系統資料安全。  

• 訂單管理功能  
使用者可建立新訂單、查詢既有訂單、修改訂單內容，系統會依訂單狀態更新物流流程。  

• 庫存管理功能  
系統可記錄商品入庫與出庫資訊，並即時更新庫存數量，避免庫存資料錯誤。  

• 配送管理功能  
配送人員可回報配送狀態，系統將同步更新物流資訊，供管理人員查詢。  

• 報表分析功能  
系統依據訂單、庫存與配送資料，自動產生統計報表，輔助管理決策。

## 四、測試計畫   
### 4.1 測試目標    
本測試計畫之主要目標在於驗證物流資訊管理系統是否符合需求管理說明書（SRS）所定義之功能與非功能需求，確保系統在實際使用情境下能穩定、正確地運作。透過系統化測試流程，降低系統錯誤風險並提升整體品質。  

具體測試目標說明如下：  

• 確認各功能模組（訂單、庫存、配送等）能依設計規格正常運作。  
• 驗證模組間之資料傳遞與流程整合正確無誤。  
• 確保不同使用者角色之權限控管機制正確。  
• 檢測系統於多筆資料操作下之穩定性與效能。  
• 提供測試結果作為系統修正與改善之依據。  

### 4.2測試類型  

為確保物流資訊管理系統之功能正確性與系統穩定性，本測試計畫依系統開發流程規劃下列測試類型：  

• 單元測試（Unit Test）  
針對各功能模組進行個別測試，確認模組內之功能是否能依設計規格正確運作，例如訂單建立、庫存更新等功能。  

• 整合測試（Integration Test）  
測試各模組之間的資料傳遞與流程整合情形，確保模組整合後能正常運作，例如訂單模組與庫存模組之間的資料同步。  

• 系統測試（System Test）  
以整體系統為測試對象，模擬實際使用情境，驗證系統功能、效能與穩定性是否符合需求規範。  

• 使用者測試（User Acceptance Test, UAT）  
由實際使用者進行操作測試，確認系統操作流程是否符合使用者需求，介面是否易於使用。  

## 五、結論  

本物流資訊管理系統透過系統化之需求分析、概要設計、詳細設計與測試規劃，成功建構一套具備訂單管理、庫存管理與物流配送追蹤功能之整合式資訊系統。系統以提升物流作業效率、降低人為錯誤及強化管理效能為核心目標，有效改善傳統物流管理流程中資訊分散與即時性不足的問題。  

在系統設計方面，採用模組化與分層式架構，使各功能模組職責明確，提升系統之維護性與擴充性；透過完整的測試計畫，驗證系統功能之正確性與穩定性，確保系統能符合實際使用需求。
